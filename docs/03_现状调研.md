# 开发全流程自动化对标与落地清单（2026-03-03）

## 1. 评估口径（你的目标）

目标定义：**用户只创建任务（Issue/To do），其余流程全自动闭环**。

闭环范围：
1. 自动拉起编码会话
2. 自动质量门禁（lint/test/冲突检测）
3. 自动推送分支
4. 自动创建并关联 PR
5. 自动触发代码评审（带 diff 上下文）
6. 自动状态回写（In progress / In review / Done）
7. 评审通过后自动合并（受分支保护规则约束）
8. 失败补偿与重试（幂等）
9. 全链路可观测（日志/指标/trace）
10. 无人工命令依赖（除创建任务外）

## 3. 关键差距（对你目标的 P0）

四项目共同缺口：
1. **自动建 PR 与自动合并未形成默认闭环**（多数保留人工 gate）
2. **评审触发与审查上下文注入不足**（缺少“PR diff 驱动评审”的强约束）
3. **补偿恢复能力弱**（进程重启、网络失败、重复触发下幂等不足）
4. **“只建任务即可”仍依赖手工命令**（如 `review`、`pr`、`merge`）

## 4. 借鉴策略（按层抽取，而非整套照搬）

### A. 任务与会话编排层（借鉴 Fulcrum/Grove）
- 借鉴点：任务模型、worktree + terminal/session 的可视化与操作体验
- 你应保留：**Dispatcher 作为唯一编排中枢**（防止分散在 hooks/UI）

### B. Git/PR 工具层（借鉴 Grove + Fulcrum）
- 借鉴点：commit/rebase/merge 操作的错误处理与冲突提示
- 你应增强：
  - PR 创建幂等（按 head/base 查重）
  - merge 失败分型（405/409/422）+ 自动补偿

### C. 治理与规则层（借鉴 Forge）
- 借鉴点：流程规则清晰（Wish→Forge→Review）与审计意识
- 不直接采用：**“Humans are merge gate”硬限制**（与你的零人工目标冲突）
- 替代做法：
  - 通过 GitHub 分支保护 + 状态检查替代人工 gate
  - 让自动 merge 在规则满足时执行

### D. 观测与运维层（借鉴 Fulcrum）
- 借鉴点：监控服务、状态轮询、通知机制
- 你应增强：
  - trace_id 全流程贯穿（Issue→Session→PR→Merge）
  - 指标：成功率、重试次数、各阶段耗时

## 5. 你的落地路线图（建议）

### P0（本周可完成）

3. **审查 prompt 注入**：PR URL + diff stat + diff 命令

## 6. 立即可执行的验收标准（Definition of Done）

1. 用户仅创建 To do 任务，不再手工执行 `pr/review/merge` 命令
2. 任务在 30 分钟内（示例 SLA）自动走完到 merged（正常路径）
3. 任一节点失败可在 3 次内自动补偿恢复；超限可观测告警
4. 全链路可追踪：任一任务可通过 trace_id 回溯到 PR 与 merge 结果
5. 无重复 PR、无重复 merge、无状态抖动（幂等保证）

## 9. 深度分析：四个项目在关键点上的核心设计与技术选型

本节关注“为什么它们能做到当前水平、又为什么没做到零人工闭环”。

### 9.1 Fulcrum（工程化最完整，自动化链路最接近）

#### 核心设计
1. **前后端一体的任务中台**
  - 前端：React + TanStack Router + React Query + xterm 终端视图
  - 后端：Hono + SQLite + Drizzle + WebSocket + PTY
2. **状态副作用集中化**
  - `task-status` 作为状态更新入口，统一处理广播、通知、进程收尾、worktree 创建等副作用。
3. **插件化 Agent 联动**
  - 通过 Claude/OpenCode 插件命令同步任务状态，具备“接近自动”的体验。
4. **PR 监控补全任务状态**
  - `pr-monitor` 轮询 GitHub PR 状态，PR merged 后回写任务 Done。

#### 技术选型优点
- **中小团队落地快**：SQLite + Bun/Hono 成本低，端到端可控。
- **可观测能力内建**：监控、消息通道、调度服务均在后端常驻。
- **MCP 工具体系完整**：任务、项目、仓库、部署等工具面广。

#### 对你目标的短板
- 仍保留显式命令路径（如手动 `current-task pr/review/done`）。
- PR 创建/关联并非默认全自动闭环。
- merge 自动化并非强默认，更多是“可操作能力”。

---

### 9.2 Enhanced Tmux-Orchestrator（流程框架强，落地实现证据弱）

#### 核心设计
1. **组织架构驱动流程**（Master/Coordinator/Manager 层级）。
2. **5 阶段规范化流程**（Plan/Requirements/Design/Tasks/Execute）。
3. **强调 GitHub 与看板联动叙事**。

#### 技术选型特征
- 倾向“流程声明 + 指挥层”模型。
- 适合治理与方法学导入。

#### 对你目标的短板
- 可验证代码证据不足，自动化闭环细节不清晰。
- 更像“方法论/编排壳”，不是可直接复用的自动执行内核。

---

### 9.3 Forge（治理体系与规则密度最高，但强调人工 merge gate）

#### 核心设计
1. **Wish → Forge → Review 三段式**，强过程治理。
2. **一任务一 PR、一 worktree**，并行边界清晰。
3. **大量 Git hooks/脚本自动化**（预提交、预推送、任务链接、校验）。
4. **MCP-first 与 Agent 分工明确**（谁能编排、谁能执行、谁能审查）。

#### 技术选型优点
- **规则强约束**，适合多人协作与审计留痕。
- **自动化颗粒度细**，尤其在开发流程和发布链路。

#### 对你目标的短板
- 文档中明确 “Humans are merge gate / Agents NEVER merge”。
- 价值观是“人类把关”，并非“无人值守闭环”。

---

### 9.4 Grove（Git 操作内核扎实，任务闭环工具强，但默认仍需显式触发）

#### 核心设计
1. **Rust 内核 + Web/TUI 双前端**，共享任务操作层（`operations::tasks`）。
2. **任务生命周期原子操作清晰**：commit/sync/rebase/merge/archive。
3. **MCP 工具 `grove_complete_task`** 提供“一键完成”能力。
4. **冲突处理路径明确**：rebase 冲突识别、回滚、重试建议。

#### 技术选型优点
- **操作可靠性高**：错误处理、回滚、状态一致性设计扎实。
- **可维护性好**：业务逻辑与 UI 解耦明显。

#### 对你目标的短板
- `grove_complete_task` 被明确限制为“用户显式要求时调用”。
- Merge UI 和流程仍以人工触发为主，不是事件驱动自治状态机。

## 10. 与 multi-agent-workflow 的架构对照

### 10.1 你当前架构的独特优势

1. **Central Dispatcher 天然适配“零人工闭环”**
  - 以状态机和补偿机制为核心，天然比“UI/插件驱动”更容易做到无人值守。
2. **VK + GitHub 双通道编排**
  - 看板状态驱动会话，PR 驱动审查与合并，链路更清晰。
3. **可最小侵入接入任意项目**
  - 通过模板与 hooks 注入，不需替换业务系统。

### 10.2 相比优秀项目仍需补强

1. **产品化体验层**（对标 Fulcrum/Grove）
  - 当前偏“调度内核”，缺少统一可视化控制面和运行态诊断视图。
2. **Git 操作故障模型**（对标 Grove）
  - 需进一步标准化冲突分类、自动回滚策略、可恢复提示。
3. **流程治理文档与审计规范**（对标 Forge）
  - 需补齐策略规范与执行证据模板，便于团队规模化。

## 11. 我最认可的“优秀项目”与理由

### 结论：**Fulcrum**（综合工程成熟度第一）

> 说明：该结论是“综合工程能力最佳”，不是“最符合零人工闭环目标”。
> 对“零人工闭环”的目标匹配度，仍需要你现在的 Dispatcher 思路补齐。

### 11.1 选择 Fulcrum 的具体理由

1. **系统完整度最高**
  - 前端控制台、后端服务、CLI、插件、MCP 工具链、监控与通知体系基本齐全。
2. **状态驱动的工程实践成熟**
  - 状态更新副作用集中处理，减少散落逻辑。
3. **对外联动能力强**
  - GitHub PR 监控、消息通道、调度服务均有落地实现。
4. **可扩展性优于“纯脚本编排”**
  - 在多功能扩展（部署、消息、观察者）上展示了较强演进能力。

### 11.2 为什么不是 Forge / Grove / Enhanced

- **不是 Forge**：规则设计非常强，但明确保留人工 merge gate，与“零人工”目标冲突。
- **不是 Grove**：Git 操作内核优秀，但默认仍是“显式触发完成”，不是自治调度。
- **不是 Enhanced**：流程叙事完整，实际可验证实现密度不足。

### 11.3 对你最有价值的“Fulcrum 借鉴包”

建议按以下顺序迁移思想，而非迁移代码：

1. **状态副作用集中化规范**
  - 将 Dispatcher 的动作与副作用边界文档化（状态更新、通知、会话、PR）。
2. **运行态监控面板模型**
  - 最少要有：当前队列、失败重试、平均耗时、补偿次数、最近异常。
3. **工具边界分层**
  - 编排层（Dispatcher）/执行层（hooks、agent）/可视化层（状态展示）职责明确。
