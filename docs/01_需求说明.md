## 1. 背景与目标

### 1.1 背景

现代软件开发中，单人或小团队面临以下挑战：

- 需求到代码的链路长，人工衔接成本高
- 多任务并行时上下文切换频繁，精力碎片化
- Code Review 依赖他人响应，存在等待瓶颈
- 从代码完成到合并上线，仍有大量手工操作
- 跨 AI Agent 工具体系缺少统一调度，各自为政

### 1.2 核心目标

构建一套**多 Agent 并行开发工作流模板**，实现：

> **用户只需创建任务，其余全部自动化执行**

具体目标：
1. 一条命令初始化任意项目，注入完整工作流能力
2. 多个 AI Agent（Claude Code / Codex CLI / Gemini CLI）在独立 Git Worktree 中并行开发
3. 代码完成后自动创建 PR、触发交叉审查、合并到主分支
4. 全流程可观测，状态可追踪，异常可自动补偿

## 2. 用户故事

### US-001 一键初始化
**As** 开发者  
**I want to** 对任何现有或新项目执行一条命令就完成工作流注入  
**So that** 不需要手动配置每个工具和脚本  

验收标准：
- `bash init.sh -p <project>` 成功执行，生成所有必要文件
- 对已有项目不覆盖现有文件，只追加缺失内容
- 非交互模式 (`--non-interactive`) 生成 TODO 占位符供事后填写

### US-002 任务驱动自动化
**As** 开发者  
**I want to** 只在看板上创建 Issue，后续编码、PR、Review、合并均由系统自动完成  
**So that** 解放精力专注于需求本身而非流程操作  

验收标准：
- Issue 状态「To do」→「In progress」可自动触发编码 Session 创建（可开关）
- Issue 状态「In progress」完成后 → 自动创建 GitHub PR
- PR 创建后 → 自动触发交叉审查 Session
- Issue 状态「Done」→ 自动合并 PR 到主分支

### US-003 多 Agent 并行隔离
**As** 开发者  
**I want to** 多个 AI Agent 同时处理不同任务而互不干扰  
**So that** 开发效率线性提升，不出现代码冲突  

验收标准：
- 每个任务在独立 Git Worktree 中执行
- 各 Agent 不共享工作目录
- 冲突检测脚本在合并前执行，有冲突则阻断并提示

### US-004 交叉审查
**As** 开发者  
**I want to** 代码自动被不同类型的 Agent 审查  
**So that** 发现编码 Agent 遗漏的问题，提升代码质量  

验收标准：
- 编码 Agent 与审查 Agent 必须为不同类型（Claude/Codex/Gemini）
- 审查 Agent 注入 PR diff 作为上下文
- 审查结论记录到 PR 评论或 VK 报告

### US-005 质量门禁
**As** 开发者  
**I want to** 每次代码提交前自动运行 lint 和测试  
**So that** 低质量代码不进入主分支  

验收标准：
- Agent 完成编码后，自动执行 lint + 单元测试
- 全部通过后才 push 分支、变更 Issue 状态
- 测试失败则阻断流程，记录失败原因

### US-006 补偿机制
**As** 开发者  
**I want to** 调度器在中断后重启时能自动恢复未完成的操作  
**So that** 无需人工识别并重新触发失败的步骤  

验收标准：
- 调度器重启后检测 Issue 当前状态与已执行动作
- 「In review 但无 PR」→ 自动补偿创建 PR
- 「Done 但 PR 未合并」→ 自动补偿 merge

### US-007 可观测性
**As** 开发者  
**I want to** 随时查看当前各任务状态和调度器运行情况  
**So that** 快速定位异常、了解进度  

验收标准：
- 每次轮询生成唯一 trace_id
- 结构化日志包含：时间、级别、trace_id、操作摘要
- 累计指标可查：轮询次数、动作数、错误数

## 4. 非功能需求

### 4.1 可用性
- 零外部依赖（纯 Python 标准库实现）
- 支持 macOS / Linux / WSL 环境
- Python 3.10+

### 4.2 可靠性
- 调度器异常退出后，重启能自动从持久化状态恢复
- 网络请求设置超时，失败不崩溃，记录日志继续运行
- 幂等设计：同一 Issue 的同一动作不重复执行

### 4.3 可观测性
- 每次轮询生成 6 位十六进制 trace_id，贯穿该轮所有日志
- 结构化日志格式：`[时间] [级别] [trace] 消息`
- 累计指标：poll_count、action_count、error_count

### 4.4 可配置性
- 所有自动化开关可通过 `.vk/dispatcher.json` 独立开关
- 轮询间隔、合并方式、Agent 分配规则均可配置
- 支持环境变量覆盖关键配置（VK_PORT 等）

### 4.5 安全性
- GitHub Token 不写入 git 提交（`.vk/github_token` 在 `.gitignore` 中）
- 敏感配置仅读取自环境变量或本地文件

### 4.6 可扩展性
- Agent 类型可通过配置扩展（cross_review_map）
- PR 模板支持自定义
- 质量门禁命令可逐项目定制

## 6. 用户界面需求

本项目为开发者工具，不含 GUI。交互方式：

1. **命令行**：`bash init.sh`、`python -m dispatcher`、`make dispatcher`
2. **Vibe Kanban** 看板：浏览器访问任务状态（由 VK 提供，非本项目实现）
3. **VS Code Copilot**：Plan Mode 中通过 MCP 创建任务、查看状态
4. **日志输出**：结构化日志到 stdout，便于接入日志收集系统

## 7. 验收标准汇总

| 场景 | 验收标准 |
|------|----------|
| 初始化新项目 | `bash init.sh -p test` 执行无报错，生成所有文件 |
| 注入已有项目 | 已有文件无覆盖，缺失文件正常生成 |
| 调度器启动 | VK 不可达时提示错误退出，可达时正常轮询 |
| 状态变化响应 | Issue 转 In review 后 60s 内创建 PR 和审查 Session |
| 补偿恢复 | 调度器重启后自动执行遗漏的 PR 创建或 merge |
| 质量门禁 | 测试失败时 Issue 状态不变更，分支不 push |
| dry-run | 有状态变化时输出 DRY-RUN 日志，无实际操作 |
| 幂等性 | 同一 Issue 多次轮询到同一状态，不重复执行动作 |
