## 1. 架构总览

### 1.1 设计哲学

- **最小侵入**：通过模板注入，无需修改目标项目的业务逻辑
- **零依赖**：调度器纯 Python 标准库实现，无需安装额外包
- **状态机驱动**：所有自动化以 VK Issue 状态变迁为唯一触发源
- **补偿优先**：每轮轮询检查未完成动作，崩溃恢复后自动续跑
- **可观测性内建**：每轮附 trace_id，指标累积，日志结构化

## 3. 核心状态机

```
                  ┌───────────┐
                  │  To do    │
                  └─────┬─────┘
                        │ auto_start_coding = true
                        │ 创建编码 Session（VK MCP）
                        ▼
                  ┌───────────┐
                  │In progress│  ←── Agent 正在编码
                  └─────┬─────┘
                        │ vk-hooks.sh: lint+test 通过
                        │ push → cleanup_script 完成
                        │ Dispatcher 轮询 SQLite → QG passed
                        ▼
                  ┌───────────┐
   Dispatcher:    │ In review │
   ① 状态流转 ───►│           │◄── Dispatcher 检测到 QG 通过后触发
   ② create_pr    └─────┬─────┘
   ③ start_review
                        │ 审查 Agent 审查通过
                        │ VK 状态更新 → Done
                        ▼
                  ┌───────────┐
   Dispatcher:    │   Done    │
   merge PR ─────►│           │
                  └───────────┘
```

### 状态转换条件一览

| 源状态 | 目标状态 | 触发者 | 前置条件 | 调度器动作 |
|--------|----------|--------|----------|------------|
| To do | In progress | Dispatcher | `auto_start_coding=true` | 创建编码 Session |
| In progress | In review | Dispatcher | SQLite: QG 通过（cleanup=completed 或 codingagent=completed） | ① 状态→In review ② 创建 PR ③ 创建审查 Session |
| In review | Done | 审查 Agent | 审查通过 | merge PR |
| Done | — | — | PR 已合并 | 无 |

### QG 通过检测逻辑（REST + SQLite 混合）

Dispatcher 在每轮补偿检查中调用 `VKDatabase.is_qg_passed(branch)` 判断编码是否完成：

```
┌── repo 配置了 cleanup_script ？──────────────────────┐
│  YES                                NO               │
│  检测 cleanupscript 进程             检测 codingagent 进程 │
│  status='completed' AND exit_code=0  status='completed' AND exit_code=0 │
└──────────────────────────────────────────────────────┘
         ↓ True                              ↓ True
   Dispatcher._action_finish_coding() → In review
```

**关键 SQL 约束**（过滤 VK 重启假失败）：
- `ep.dropped = FALSE`（排除已丢弃进程）
- `exit_code IS NOT NULL`（`NULL` = VK 重启 artifact，非真实失败）
- `ORDER BY started_at DESC LIMIT 1`（取最新一条）

**三态返回**：`True`（通过）/ `False`（真实失败，等待人工）/ `None`（运行中或未开始）

### 补偿场景

| 检测条件 | 补偿动作 |
|----------|----------|
| 状态=In progress 且有 `coding_branch` 且 SQLite QG=True | 状态→In review，创建 PR，创建审查 Session |
| 状态=In progress 且有 `coding_branch` 且 SQLite QG=False | 记录警告，等待人工介入 |
| 状态=In review 且 `pr_number=null` 且有 `coding_branch` | 创建 PR |
| 状态=In review 且 `review_workspace_id=null` | 创建审查 Session |
| 状态=Done 且 `pr_merged=false` 且有 `pr_number` | 合并 PR |

## 4. 数据模型

### 4.1 IssueTracker（调度器内存 + 持久化）

```python
@dataclass
class IssueTracker:
    status: str                      # 当前 Issue 状态
    title: str                       # Issue 标题
    simple_id: str                   # 人类可读 ID（如 PROJ-42）
    # 编码阶段
    coding_workspace_id: str | None  # VK Workspace ID
    coding_branch: str | None        # 编码分支名
    coder_executor: str | None       # 编码 Agent 类型
    # PR
    pr_number: int | None            # GitHub PR 号
    pr_url: str | None               # PR 链接
    pr_merged: bool                  # 是否已合并
    # 审查阶段
    review_workspace_id: str | None  # 审查 Session Workspace ID
    review_branch: str | None        # 审查分支名
    # 合并
    merged: bool                     # 最终是否已合并
    # 时间戳
    updated_at: str                  # 最后更新时间

# 注: v1.0 的 in_progress_since / last_qg_sha 已在 v1.1 移除。
# QG 状态改由 VKDatabase.is_qg_passed() 直读 VK SQLite 获取，无需文件标记。
```

持久化路径：`{project_dir}/.vk/dispatcher_state.json`

### 5.2 VKDatabase（SQLite 直读）

```python
class VKDatabase:
    """直读 VK 本地 SQLite（~/.local/share/vibe-kanban/db.v2.sqlite）"""

    def is_qg_passed(self, branch: str) -> bool | None:
        """检测指定分支的 QG 是否通过。

        1. 若 repo 配置了 cleanup_script → 以 cleanupscript 进程完成为准
        2. 否则 → 以 codingagent 进程成功完成为准

        返回:
          True  — QG 通过（可流转 In review）
          False — Agent 真实失败（exit_code IS NOT NULL AND exit_code != 0）
          None  — 仍在运行 / 尚未启动 / 数据不足
        """

    def is_agent_running(self, branch: str) -> bool:
        """检查编码 Agent 是否仍在运行（dropped=FALSE AND status='running'）"""

    def has_cleanup_script(self, branch: str) -> bool:
        """检查关联仓库是否配置了 cleanup_script"""

    def get_latest_process(self, branch: str, run_reason: str) -> dict | None:
        """获取指定分支、指定 run_reason 的最新执行记录"""
```

**SQLite 路径**：`~/.local/share/vibe-kanban/db.v2.sqlite`

**重要约束**：
- 所有查询必须加 `ep.dropped = FALSE`
- `exit_code IS NULL` = VK 重启 artifact（`cleanup_orphan_executions`），不视为失败
- VK remote 项目模式下 `tasks` 表为空；Issue 状态只能通过 REST API 获取

## 7. 安全设计

| 攻击面 | 防护措施 |
|--------|----------|
| GitHub Token 泄漏 | 写入 `.vk/github_token`，在 `.gitignore` 中排除 |
| VK 端口越权 | VK 绑定 127.0.0.1，仅本地访问 |
| 恶意 Agent 代码提交 | 质量门禁 `lint + test` 阻断，不通过不 push |
| PR 误合并 | `auto_merge` 默认开启但状态机严格校验 Issue 状态 |
| 重复执行 | `pr_number` / `pr_merged` 字段幂等保护 |

## 10. 技术选型决策记录（ADR）

### ADR-001：纯标准库实现调度器
- **决策**：Dispatcher 仅使用 Python 标准库（`urllib`、`subprocess`、`json`）
- **原因**：零 pip 依赖，在任意 Python 3.10+ 环境即可运行；降低注入到目标项目的侵入性
- **代价**：HTTP 客户端稍冗长，无 retry 中间件，需自行实现补偿

### ADR-002：VK 作为唯一任务编排中枢
- **决策**：所有任务状态变更通过 VK 看板，Dispatcher 被动监听
- **原因**：避免状态分裂；允许人工在看板上干预任何阶段；VK 提供 MCP 接口供 AI 直接操作
- **代价**：Dispatcher 完全依赖 VK 可用性，VK 宕机则自动化停止

### ADR-003：状态机 + 补偿优于事件 Webhook
- **决策**：使用轮询 + 幂等补偿，而非 VK Webhook 推送
- **原因**：VK 当前版本不提供 Webhook；轮询 + 持久化状态机对进程崩溃更鲁棒
- **代价**：状态变化响应有最多 `poll_interval` 秒延迟（默认 30s）

### ADR-004：GitHub squash merge 为默认合并方式
- **决策**：PR 默认使用 squash merge
- **原因**：保持主分支提交历史整洁；Agent 并行开发产生的中间提交不进入主分支
- **代价**：feature 分支的细粒度提交历史在主分支不可见
### ADR-005：用 SQLite 直读替代文件标记检测 QG 状态（v1.1）
- **决策**：删除 `.vk/qg_passed/<sha>` 文件标记系统和 QG 兜底执行器；改为直读 VK 本地 SQLite DB
- **背景**：标记文件在 Dispatcher 重启后持久化 → 造成 `In progress → In review` ping-pong；QG 兜底执行器在 worktree 中运行业务脚本破坏了"Dispatcher 不执行业务逻辑"的职责边界
- **方案对比**：
  - REST API only：`/api/remote/workspaces/{id}` 只返回快照，无进程历史，running 状态不可靠
  - SQLite only：`execution_processes` 表完整记录所有进程状态和 exit_code，是权威来源
  - **REST + SQLite hybrid（选用）**：Issue 状态用 REST（remote 模式 `tasks` 表为空），Agent 运行/完成状态用 SQLite
- **为何 SQLite 可靠**：VK cleanup_script 仅在 codingagent 正常完成（exit_code=0）后触发；`exit_code=NULL` = VK 重启 artifact（`cleanup_orphan_executions()`），非真实失败；结论来自 VK 源码分析（`crates/container.rs`、`crates/execution_process.rs`）

### ADR-006：Remote 模式下本地 SQLite 与云端 PostgreSQL 职责分离
- **决策**：在 VK Remote 项目模式下，Issue/Project/Organization 数据存于云端 PostgreSQL；Workspace 执行层数据（进程状态、Agent 日志）存于本地 SQLite。Dispatcher 跨越两层：读本地 SQLite 检测 Agent 完成状态，写云端 PostgreSQL 推进 Issue 状态
- **背景**：使用 VK Remote 集成模式时，本地 `tasks` 和 `projects` 表行数为 0，任何直读这两张表的代码都会静默失败
- **架构约束**：
  1. Issue 状态（To do / In progress / In review / Done）**只能经由 `/api/remote/issues` REST 代理**读写；VK 本地服务将请求转发至 `VK_SHARED_API_BASE`（云端，build-time 编译进二进制，可被同名环境变量覆盖）
  2. Workspace ↔ Issue 关联存于云端 `remote_workspaces` 表（`local_workspace_id → issue_id`），本地 SQLite `workspaces.task_id` 列在 remote 模式下为 NULL
  3. `coding_agent_turns.summary` 是**本地 SQLite**，也是 Dispatcher 读审查结论的唯一来源；这是唯一"本地数据驱动云端状态变更"的桥接点
  4. 认证通过 `~/.local/share/vibe-kanban/credentials.json` 中 JWT `refresh_token` 完成，无需额外配置
- **代价**：Dispatcher 必须同时依赖本地 SQLite 权限和云端网络连通性；二者任一不可用，自动化停止

## 12. 数据存储架构（Local SQLite vs Cloud PostgreSQL）

### 12.1 双存储架构概览

VK 在 Remote 项目模式下采用**本地 SQLite + 云端 PostgreSQL** 双层存储：

```
┌──────────────────────────────────────────────────────────────┐
│              本地 SQLite                                      │
│   ~/.local/share/vibe-kanban/db.v2.sqlite                    │
│                                                              │
│   [ 执行层 — Agent 运行状态 ]                                  │
│   workspaces (585)          ← Workspace 物理路径、分支名       │
│   sessions   (360)          ← 每个 workspace 的会话           │
│   execution_processes (819) ← 进程状态、exit_code  ← 权威来源  │
│   coding_agent_turns  (359) ← Agent 输出，summary 含审查结论  │
│   execution_process_logs    ← Agent 完整日志流                │
│   repos                     ← setup/cleanup_script 配置      │
│                                                              │
│   tasks    = 0 行  ┐                                         │
│   projects = 0 行  ┘ ← Remote 模式下永远为空，禁止直读         │
└──────────────────┬───────────────────────────────────────────┘
                   │ VK 本地 Rust 服务 :9527
                   │ /api/remote/* → RemoteClient 透明转发
                   │ 认证: credentials.json refresh_token JWT
                   ▼
┌──────────────────────────────────────────────────────────────┐
│              云端 PostgreSQL                                  │
│   VK_SHARED_API_BASE（build-time 编译入二进制，可被同名 env 覆盖）│
│                                                              │
│   [ 业务层 — Issue 生命周期 ]                                  │
│   organizations       ← 组织信息                             │
│   projects            ← 项目（65f08fde-...）                  │
│   issues              ← Issue 标题、状态（To do / In review…） │
│   project_statuses    ← 状态定义及 UUID 映射                  │
│   remote_workspaces   ← local_workspace_id ↔ issue_id 关联  │
└──────────────────────────────────────────────────────────────┘
```

### 12.2 数据归属速查

| 数据类型 | 存储层 | 访问方式 | 注意事项 |
|---------|-------|---------|--------|
| Issue 标题、状态 | 云端 PostgreSQL | REST `/api/remote/issues` | 本地 `tasks` 表为空，禁止直读 |
| 项目、组织信息 | 云端 PostgreSQL | REST `/api/remote/projects` | |
| Workspace ↔ Issue 关联 | 云端 PostgreSQL `remote_workspaces` | REST `by-local-id` | 本地 `workspaces.task_id` 为 NULL |
| Workspace 物理路径、分支 | 本地 SQLite `workspaces` | REST `/api/task-attempts` | |
| Agent 进程状态、exit_code | 本地 SQLite `execution_processes` | SQLite 直读 | **进程完成的权威来源** |
| **Agent 审查结论** | **本地 SQLite `coding_agent_turns.summary`** | **SQLite 直读** | **唯一跨层桥接点 ★** |
| VK Repo 配置脚本 | 本地 SQLite `repos` | SQLite 读 / MCP 写 | E-CFG 自修复依赖此表 |

### 12.3 Dispatcher 跨层调用链

```
① 读本地 SQLite → 检测 Agent 审查结论
   VKDatabase.is_review_done(review_branch)
   └─ SELECT coding_agent_turns.summary
        WHERE workspace.branch = review_branch
        ORDER BY created_at DESC LIMIT 1
   └─ 解析 'Review 结论: APPROVED / CHANGES_REQUESTED'
   └─ 返回 "approved" / "changes_requested" / False / None

② 写云端 PostgreSQL → 推进 Issue 状态
   VKRestClient.update_issue_status(issue_id, "Done")
   └─ PATCH http://localhost:9527/api/remote/issues/{id}
        └─ VK Rust 服务: deployment.remote_client()
             └─ RemoteClient.update_issue()
                  └─ PATCH {VK_SHARED_API_BASE}/api/issues/{id}
                       └─ 云端 PostgreSQL UPDATE
```

**核心结论**：`coding_agent_turns.summary`（本地 SQLite）是整个流水线中**本地执行状态 → 云端 Issue 状态**变更的唯一桥接点。正确性取决于 Agent 严格按 prompt 格式输出 `### Review 结论: APPROVED/CHANGES_REQUESTED`。

### 12.4 /api/remote/* 代理机制

VK 本地服务（`:9527`）所有 `/api/remote/*` 路由均为**纯透明代理**，无本地缓存或持久化：

```
Dispatcher/Browser              VK 本地 Rust (:9527)           VK 云端
GET  /api/remote/issues    →   issues::list_issues()      →   PostgreSQL SELECT
POST /api/remote/issues    →   issues::create_issue()     →   PostgreSQL INSERT
PATCH /api/remote/issues   →   issues::update_issue()     →   PostgreSQL UPDATE
                           ←   JSON response              ←
```

源码：`vibe-kanban/crates/server/src/routes/remote/issues.rs`
所有 handler 均调用 `deployment.remote_client()?.xxx()`，无任何本地持久化逻辑。

### 12.5 认证流程

```
VK 启动时:
  读取 ~/.local/share/vibe-kanban/credentials.json
  └─ { "refresh_token": "eyJ..." }
       └─ 向 VK_SHARED_API_BASE 换取 access_token
       └─ RemoteClient 携带 Bearer token 调用云端 API

Token 过期 → VK 自动使用 refresh_token 换新 → 无需人工介入
credentials.json 丢失 → /api/remote/* 全部 401 → Dispatcher 无法读取 Issue 状态
```
